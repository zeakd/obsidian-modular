# UX Plan — Modular (2026-05)

전체 사용자 경험 관점에서 modular plugin이 무엇을 잘하고 어디가 미흡한지,
어떤 우선순위로 발전할 수 있는지 정리.

## 1. modular의 raison d'être

> **Vault의 markdown을 "구조화된 계층 + 의존 관계 그래프"로 표현하고, 어느
> 쪽에서 편집해도 양방향으로 sync.**

핵심 3가지 가치:
1. **구조화** — Obsidian Canvas의 자유 카드와 달리 modular는 module/component
   hierarchy가 강제됨. 그래서 entity 사이 관계가 명시적
2. **양방향 sync** — Canvas에서 편집해도 vault 파일이 바뀌고, vault에서 파일을
   직접 편집해도 Canvas가 반영. 두 표현이 같은 데이터의 두 얼굴
3. **AI/automation 친화** — frontmatter `modular-id` (ULID) 기반의 stable
   reference. AI가 entity를 추적하고 변경할 수 있음

### 경쟁/유사 도구 비교

| 도구 | 데이터 | 구조 | 양방향 | 모바일 | AI |
|---|---|---|---|---|---|
| **Obsidian Canvas** (native) | .canvas 단일 파일 | 자유 | ↔ | △ | △ |
| **Excalidraw plugin** | .excalidraw | 자유 그림 | ↔ | △ | × |
| **Notion DB** | cloud | 강제 schema | ↔ | ✓ | ✓ |
| **mind-map plugins** | 단일 .md | 트리 only | △ | × | × |
| **modular** | folder + _index.md per entity | 계층 + task edge | ✓ id-based | △ | ✓ |

modular의 unique edge:
- **per-entity 파일 분리** (단일 .canvas 파일 아님) — git diff 가능, AI tool이
  파일 단위로 작업
- **id-based reference** — rename/move에 invariant
- **vault-native** — 별도 server / cloud / DB 없음

## 2. 현재 UX 평가

### 강점
- 노드 더블클릭 → 이름 입력 즉시 entity 생성 (vault 파일까지 자동)
- 노드 클릭 → side leaf 에 _index.md 자동 열림 (Canvas-Vault 동시 view)
- ID 기반이라 사용자가 vault explorer에서 자유롭게 폴더 옮겨도 task 끊기지 않음
- React Flow의 표준 UX (pan, zoom, drag, edge handle) 그대로 사용

### 미흡 — 사용자 여정별로

#### A. First touch (첫 install)
- 빈 캔버스 + 작은 hint "더블클릭으로 모듈 생성" — 정보 부족
- 무엇이 module이고 component인지, task edge가 뭔지 설명 없음
- 첫 사용자가 "어떻게 시작하지?" 시점에 멈출 위험

#### B. 자식 추가
- Tab 키 단축 — 발견 어려움 (status bar 힌트는 있지만 selectedId 있을 때만)
- 우클릭 메뉴는 있지만 "자식 추가" 라벨이 트리 구조 의도를 전달 못 함
- 모바일에선 Tab/우클릭 둘 다 어려움

#### C. Task edge 그리기
- Reactflow handle을 dragging — 표준이지만 "component 사이만 가능"이라는 제약이
  안내 X. module-module / module-component drag는 silent ignore
- edge에 label 달 수 없음 (현재 데이터 모델에 없음)

#### D. 삭제
- 네이티브 `confirm()` dialog — 모바일 / Obsidian 테마와 안 어울림
- 폴더 통째 trash 사실을 dialog가 명확히 알려주지만 그 자체가 무서움

#### E. Vault 직접 편집
- 새 `_index.md` 만들 때 frontmatter `modular-id` 필요 → 사용자가 직접 ULID 적기
  부담
- frontmatter `modular-parent` / `modular-tasks` 수동 작성 시 id 외울 수 없음

#### F. 검색 / 찾기
- 없음. entity 20-30개 넘으면 시야 밖 entity는 pan + zoom out으로만 찾음
- Obsidian quick switcher는 _index.md만 보여줘서 폴더 이름 기준 검색 어려움

#### G. 본체 내용 미노출
- _index.md의 markdown body는 frontmatter만 읽고 무시
- 사용자가 entity 안에 길게 적은 메모를 Canvas에선 못 봄
- side leaf 열어야만 본문 확인

#### H. 모바일
- Vertical split 없음 → tab fallback 추가됨 (0.2.0). 단 task edge handle 터치
  정밀도는 미검증
- 키보드 단축 (Tab / Delete / ⌘Enter) 없음
- confirm dialog가 모바일 UX 깨뜨림

#### I. Obsidian 네이티브 통합
- modular-tasks는 frontmatter list (plain string id) — Obsidian graph view는
  못 봄 (wiki-link 기반)
- backlink 패널에 incoming task 안 나옴

#### J. AI / automation
- frontmatter API 안정적이지만 SDK / docs 없음 — 외부 도구가 modular 파일을
  안전하게 변경하려면 직접 spec 읽어야

## 3. 사용자 여정 (예상 typical journey)

```
첫 install
  ↓
Settings → Modular 활성화
  ↓
ribbon "git-fork" 아이콘 또는 command "Open canvas" → 빈 Canvas
  ↓
더블클릭 → 모듈 "payments" 만듦  ← 첫 성공
  ↓
모듈 더 만들기 (5-10개) → vault에 폴더 5-10개 생김
  ↓
Tab 발견 → component 추가 (자식)
  ↓
component handle drag → task edge ← "왜 module끼리는 안 되지?" 멈춤
  ↓
편집하다 보니 entity 20+ → 화면 안 보임 → 검색 시도 → 없음 → 좌절
  ↓
모바일에서 보고 싶음 → install → component drag 어려움 → 안 씀
  ↓
며칠 후 잊음, 다시 vault explorer 보니 modular 폴더가 어색하게 남아 있음
```

각 단계의 dropout 위험을 줄이는 게 UX 작업의 핵심.

## 4. 단계별 roadmap

각 phase는 머지 가능한 단위로 더 잘게 쪼개질 수 있음. 우선순위는 "사용자
가치 / 작업량" 기준.

### Phase 1 — 첫 사용자 가드레일 (가장 시급)

**목표:** 첫 install 후 5분 내에 "modular가 뭘 하는지" 이해 + 깔끔한 액션 UX.

| 항목 | 핵심 |
|---|---|
| **1.1 Onboarding** | 빈 Canvas에 단계별 hint (Step 1 더블클릭 → Step 2 이름 → Step 3 Tab으로 자식 등). 초기 상태에서만 표시, dismiss 가능 |
| **1.2 Modal로 confirm 대체** | obsidian `Modal` 또는 third-party confirm modal. 모바일/테마 친화 |
| **1.3 사용법 hint 강화** | status bar 항상 표시 (selectedId 무관). 단축키 cheatsheet command |
| **1.4 빈 entity에 placeholder** | 새 entity의 _index.md body가 빈 경우 "여기에 이 모듈의 설명을 적어보세요" 같은 hint comment |

### Phase 2 — Vault native 통합 (가장 큰 임팩트)

**목표:** modular가 Obsidian "위에 얹은 plugin"이 아니라 "Obsidian 흐름의
자연스러운 일부"가 되도록.

| 항목 | 핵심 |
|---|---|
| **2.1 Wiki-link 통합** | `modular-tasks` 값을 `[[modular/payments/_index]]` 형식 wiki-link 로 저장. obsidian graph view / backlinks / quick switcher에서 자동 인식. id 기반 ref와 link ref 양립 |
| **2.2 자동 id 부여** | `modular/` 하위 `_index.md` create 시 frontmatter에 id 없으면 자동 patch. 사용자가 vault에서 직접 entity 만들기 쉬워짐 |
| **2.3 검색** | `Modular: Find entity` command. 이름 기반 quick switcher 스타일. 결과 클릭 = Canvas에서 해당 노드로 zoom + select |
| **2.4 폴더-frontmatter divergence 경고** | rebuild 시 `modular-parent` ≠ 폴더 nesting이면 Notice 또는 settings에 표시 |

### Phase 3 — Canvas 작업 효율 (파워 유저)

**목표:** 정말 사용하는 사람의 일일 효율.

| 항목 | 핵심 |
|---|---|
| **3.1 Drag로 부모 변경** | 노드를 다른 노드 위로 drag stop → confirm Modal → 폴더 이동 + frontmatter parent 갱신 |
| **3.2 본체 markdown preview** | 노드 hover 또는 zoom-in 시 _index.md body의 첫 N라인 표시. side leaf 안 열고도 내용 일별 |
| **3.3 Task edge label** | edge 더블클릭 → 라벨 편집. frontmatter `modular-tasks`를 `[{to: id, label: "..."}]` 객체 array로 확장 |
| **3.4 Bulk operations** | 다중 선택 후 일괄 tag 추가/삭제, 일괄 부모 변경 |
| **3.5 필터** | tag별 hide/show, kind별 (module만/component만) |

### Phase 4 — 모바일 first-class

**목표:** 모바일이 "데스크탑의 viewer"가 아니라 "실제 작업 환경" 되도록.

| 항목 | 핵심 |
|---|---|
| **4.1 터치 UX 검증** | reactflow drag/pinch/edge handle을 실측. 필요시 큰 hit target |
| **4.2 모바일 단축 UI** | Tab/Delete/⌘Enter 대신 컨텍스트 bottom sheet (자식 추가 / 삭제 / 부모 변경) |
| **4.3 split → tab fallback 다듬기** | 0.2.0 fallback 동작 확인 + UX 정돈 (swipe back 안내) |

### Phase 5 — AI / automation API

**목표:** 외부 AI 도구가 modular 파일을 안전하게 다루도록 SDK 제공.

| 항목 | 핵심 |
|---|---|
| **5.1 modular-api 패키지** | `obsidian-modular-api` npm — entity CRUD, 검색, 그래프 traversal. frontmatter 직접 다루지 않고 안전한 wrapper |
| **5.2 LLM 추출 helper** | "이 markdown body에서 task로 표현될 수 있는 항목 추출" 같은 helper. 사용자 vault에 LLM 결과를 modular-tasks로 자동 반영 |
| **5.3 변경 history** | git log 기반 entity 변경 history 패널 |

### Phase 6 — 다중 시각화

**목표:** 같은 데이터, 여러 perspective.

| 항목 | 핵심 |
|---|---|
| **6.1 Tree view** | 전통적 폴더 트리 — 큰 vault에서 hierarchy 빠르게 |
| **6.2 Kanban** | state 컬럼 (todo/doing/done) 추가 후 칸반 view |
| **6.3 Timeline** | `modular-id` ULID의 시간 정렬 활용 → entity 생성 timeline |
| **6.4 가독성 옵션** | entity 색상, 아이콘, 형태 customization |

### Phase 7 — 데이터 모델 확장

**목표:** 더 풍부한 도메인 표현.

| 항목 | 핵심 |
|---|---|
| **7.1 entity state** | `modular-state: todo/doing/done`. 칸반/timeline 연동 |
| **7.2 task 메타** | priority / due / weight |
| **7.3 entity type 확장** | `decision`, `risk`, `idea` 같은 kind |

## 5. 우선순위 추천

각 phase는 직교적 가치 — 동시 진행 가능. 단순 추천:

```
지금: Phase 1 (가드레일) — self-use에서도 자기 자신이 첫 사용자
다음: Phase 2 (vault 통합) — 가장 큰 임팩트, modular의 정체성 강화
그 다음: Phase 3 (효율) — daily user 등장 시점에 자연스러움
모바일: Phase 4는 모바일 첫 install 시도 후 평가
SDK: Phase 5는 AI 도구가 등장하면 push될 것
시각화/확장: Phase 6/7은 use case 확보 후
```

## 6. 결정 필요

핵심 질문:

### Q1. 가장 큰 사용자가 누구인가?
- (a) 본인 (self-use, dogfood)
- (b) Obsidian 사용자 일반 (community plugin 제출 목표)
- (c) AI 도구가 다루는 데이터 모델 (modular = AI 친화 substrate)

답에 따라 우선순위 다름:
- (a) → Phase 3 (효율) + Phase 6 (시각화)
- (b) → Phase 1 (가드레일) + Phase 4 (모바일)
- (c) → Phase 5 (SDK) + Phase 2 (vault 통합)

### Q2. 핵심 use case는?
- (i) **knowledge base hierarchy** (Wiki 대안)
- (ii) **task / project graph** (Linear/Asana 대안)
- (iii) **system / architecture diagram** (C4 model 같은)
- (iv) **AI agent 의 작업 공간** (agent가 task graph를 직접 편집)

답에 따라 Phase 6/7 (시각화/모델 확장) 방향 결정.

### Q3. 모바일 priority?
- (a) 데스크탑 first, 모바일은 viewer 정도
- (b) 모바일도 first-class

답에 따라 Phase 4 우선순위.

이 세 질문에 답하면 다음 PR의 명확한 타깃이 잡힘.
